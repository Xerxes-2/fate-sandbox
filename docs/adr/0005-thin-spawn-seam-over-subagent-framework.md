# Thin engine-owned spawn seam instead of a subagent framework

Backstage candidate generation (parallel-line / offscreen world motion) needs a **hermetic subagent**: a process that ingests hidden state (unrevealed `campaignSecrets`, secret `actorGoals`) and emits one _reviewed-before-landing_ structured candidate, while being physically unable to touch canonical state or leak secrets to the player. We surveyed the pi ecosystem for an off-the-shelf substrate and evaluated, with running spikes where warranted, `pi-subagents` (the original, in-process — limited), `@gotgenes/pi-subagents`/`pi-permission-system` (in-process, **rejected** at runtime: injection failed, `lookup` unavailable, its firewall is really a pi-agent-router companion), `@tintinweb` (not migrated), `pi-crew` (CrewAI-style dev orchestrator, ~70K lines, wrong target), `@vigolium/piolium` (17-phase security auditor whose subagents are deliberately **tool-bearing** — the opposite of a firewall), `@mjasnikovs/pi-task` (spec orchestration), and `@llblab/pi-actors` (the only general-purpose actor core — spiked to a 5/5 capability map, then **not adopted**). The decision: **own a ~100-line spawn+harvest seam in the engine, not a framework.** `run_parallel_line` assembles the full director prompt (persona + safe projection + `assembleParallelLineInput` with `privateFacts`) and the engine itself `child_process.spawn("pi", ["-p","--no-tools","--no-approve","--no-context-files","--session-dir",BACKSTAGE_SESSION_DIR,"--session-id",runId, prompt])` detached+unref'd, returning a `run_id` immediately; the next turn harvests the child's session transcript and runs `parseParallelLineOutput` as a return-trip gate before anything lands.

The rationale is structural, and it is the same lesson as ADR 0003 read from the other side: **the hard part of this feature is the domain invariants, and no generic orchestration framework encodes them.** The evaluated multi-agent extensions are primarily built for coding agents. Their subagents read files, run shell commands, scan code, and edit worktrees. They include scheduling, run registries, mailboxes, concurrency quotas, and phase DAGs that this path does not use. Their tool-bearing children also conflict with two project requirements: "引擎即真理" (the engine is the single source of truth, ADR 0001/0003) and the secret firewall (secrets must never reach the player/renderer). Adopting one of these frameworks would use a small part of its orchestration surface while requiring additional restrictions around the remaining runtime. The thin seam inverts that ratio: the correctness spine already lives in the engine (the Phase-3 obligation/review ledgers in `backstageObligations`/`backstageReviewLog`/`backstagePressure`, which are **substrate-agnostic** — `measureBackstageLedger` audits landing discipline regardless of how the candidate was produced), so the substrate only has to do the one thing frameworks make _harder_: stay hermetic. `@mjasnikovs/pi-task`, although built for another purpose, uses the same basic pattern: the engine spawns a child `pi`, the parent reads distilled output, and tests cover fake spawn and leaked tool calls. This provided another implementation example for the chosen seam.

The firewall is enforced by **construction at the spawn site, not by trust**, and any future substrate swap must preserve every invariant: process isolation (separate `pi` process, separate session); `--no-tools` (zero tool calls — the child physically cannot invoke a domain tool or write canon); `--no-approve --no-context-files` (the child loads no project extension, `AGENTS.md`, or settings, so it carries no path back to canonical state); reviewed-before-landing (the parent's `parseParallelLineOutput` + ledger is the only way a candidate becomes an offscreen event); and secrets-at-rest containment (the child's transcript necessarily holds `privateFacts`, so `--session-dir` points at `.pi/agent/backstage-sessions`, which is gitignored and co-located with `auth.json` under the game's `PI_CODING_AGENT_DIR` project isolation). Note the firewall is **not** "the subagent has no secrets" — by design it gets the full hidden knowledge; safety comes from process isolation + zero tools + inability to write canon + post-hoc review, which is exactly why in-process frameworks (`@gotgenes`) cannot provide it and a spawned process can.

The spikes also tested three extensions of the seam: persistent-memory director = pin `--session-id` per faction so the child resumes its own transcript (validated: codeword survived across independent spawns); swarm = fork N children concurrently (validated: two directors ran with zero cross-session contamination, distinct codewords); cross-faction coordination = the engine threads one director's `ordersToAllies` into another's next prompt, with the GM/canonical-state as the only shared board (validated: order changed the recipient's behavior while both stayed isolated). Each spike reused the existing seam with a small amount of code. Production use would still require lifecycle, error handling, and model-configuration tests. `@llblab/pi-actors` remains spiked-and-archived (its one unique asset, a run-inspection registry, is the only reason to revisit it — for observability, never for the firewall). For this module, a small engine-owned seam keeps the domain invariants in one place and avoids adopting scheduling features the runtime does not use.

> **Amendment (2026-07-09):** the `--model` pin (`BACKSTAGE_MODEL` = deepseek-v4-pro, originally motivated by an inherited-Opus billing failure in the spike) was dropped. Call volume is too low to justify a separate cheap model and its extra API-key setup, and anthropic credentials now live in the game's `.pi/agent/auth.json`, so the director child simply inherits the settings `defaultModel` (the main model). None of the firewall invariants above are affected. Re-pin via `--model` in `backstage-spawn.ts` if provider/billing failures reappear in the spawn logs.
