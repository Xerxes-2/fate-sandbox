实验性的 agent 驱动叙事游戏。名字还没想好，暂时叫 `fate-sandbox`。重点其实不在 Fate。

目前已测试 FSN 绫香的 Saber 开局，以及 EXTRA 岸波白野开局。开局世界线现在有 13 条可选：五战、四战、hollow ataraxia、strange Fake、Apocrypha、EXTRA/CCC、事件簿，加上月姬、空之境界、魔法使之夜和自定义。都是沙盒，只给世界结构和设定底盘，不锁原作剧情。

## 结构

底层用了 Pi Coding Agent 提供的基础设施：模型接入、Agent loop、session tree。

我在上面加了几层：知识库和字典工具，状态引擎，领域/场景/基础工具，子代理，上下文编排引擎。

普通聊天卡主要靠 prompt 和上下文撑住连续性。这里把一部分连续性拆出来，变成可读、可查、可校验的状态。代价：重，麻烦，调试起来像写一个小 runtime。换来的是模型少了一大块凭空圆的空间。

## 和 MVU 的区别

中文酒馆角色卡习惯使用 MVU 作为状态系统，我的游戏和 MVU 的区别主要在这几块：

### 状态为先

绝大部分酒馆角色卡里，模型一般是先叙事，然后根据叙事内容对 MVU 变量进行 Json Patch。

我这里是反过来的。Agent 应当先通过工具调用改变状态，然后再根据调用结果进行叙事。

工具调用结果就是叙事素材。受伤已经记录了，钱已经花了，scene beat 已经结束了，后台行动已经进了 offscreen event，模型再把这些结果写成玩家能读的正文。状态漂移被压到很低，模型有据可循地叙事，不靠口胡。

### 领域/宏工具

MVU 系统中，模型改变状态的手段一般只有 Json Patch，低效，易错。

在 Fate 沙盒里，模型接触到的是领域工具。场景变化走 scene 工具，受伤走 condition，花钱走 economy，幕后行动走 offscreen event。模型只需要说明谁、因为什么、发生了什么、玩家能看到什么痕迹，底层再把这些输入收束成状态变更。

这个粒度比 Json Patch 高很多。模型不用猜 `/actors/ayaka/conditions/3` 这种路径，也不用手写一串互相关联的 patch。一次过夜可能同时推进时间、结算供魔、关闭 scene beat、触发后台行动；用 Json Patch 写，漏一个变量就开始漂。领域工具把这些变化用一个工具事务化，顺带检查当前状态允不允许这么改。

### 内部校验

变量卡玩得多的会注意到，状态漂移和错位躲不掉。模型经常会通过 Json Patch 把值写到不该写的位置，或者写出一个看上去像状态、实际没法继续结算的东西。

Fate 沙盒里，工具入口比较宽容，允许模型用各种形状的输入描述意图。核心引擎很严，所有东西进状态前都会被校验。

钱包 id 错了会打回。当前没有打开的 scene beat，就不能强行 finish。不存在的 actor 不能凭空受伤。玩家还不知道的真名不能直接写进 public。后台事件必须有 actor / faction / location / consequence，不能只写“有人监视”。

我更想要这种失败。它会在当轮暴露问题，而不是十几轮之后变成一坨谁也说不清的幽灵设定。

### Agent loop/自我纠正

变量卡还有一个问题：模型写了一段很不错的正文，结果更新 MVU 变量时 patch 错了。大部分人会舍不得重 roll，最后自己手动修变量。

Agent loop 的体验不一样。工具报错之后，模型还有下一次行动机会。错误信息如果写得足够具体，它可以自己修。

比如钱包 id 写错，报错会列出当前可用的钱包。scene objective 为空，报错会告诉它当前场景目标是什么。beat 没结束就推进场景，工具会提醒它先收口当前 beat。

所以我这里反而欢迎模型犯一部分“可解释的错”。每犯一次，就能把工具边界、错误提示、回归测试补硬一点。最怕的是模型绕过工具直接写正文，那系统就退回普通聊天卡。

## 子代理：后台导演组

我现在用了几个项目内的子代理。它们不和玩家直接对话，也不扮演 NPC。

它们更像后台导演组：只给主 Agent 提醒、候选、审计结果，不直接写玩家看到的正文。

比如：

- `parallel-line`：玩家休息、赶路、潜行、过夜时，给一条幕后线候选，给主代理提供后续暗线叙事建议。
- `timeline-showrunner`：审计员，盯当前故事符不符合世界观主线，剧情有没有停滞，主 Agent 有没有滥用悬疑要素。

总体规则是：后台钩子必须是可交互的。

新闻可以有，但新闻只是投影，不能顶替事件本体。警察巡逻也可以有，但单纯巡逻增加撑不起剧情推进。门响、信件、旧物同理：只要能归属到某个 actor / faction / location / consequence，并且给玩家调查、规避、接触、准备或撤退的窗口，就可以用；做不到的话，它就只是不可行动的气氛钩子，应该删掉或停用。

我想要的后台线很朴素：有人为了自己的目标做了事，这件事留下痕迹，玩家可以选择追、不追、绕开、利用，或者以后承担代价。

## 最近的迭代

**结算与渲染分家。** 原本正文是 agent 回合末尾顺手写的。问题在于工具纪律和文笔在同一次生成里互相拖累，审计还抓到过真名从工作记忆漏进正文。现在每回合分两遍：结算遍只跑工具和裁定，最后提交一个结构化的 direction packet；渲染遍是一次干净的 stream 调用，只看 packet 和自己之前写过的正文。packet 过防火墙，密设定整包拦下。引擎打回也不再被玩家看到：结算端在幕后自己重试，玩家只读最终正文。渲染模型可以单独配（`FATE_RENDER_MODEL`），我目前跑 GPT-5.5 结算 + Fable 5 渲染。

**GM 纪律账本化。** prompt 不是防线：压缩之后它就没了，也没法审计。所以把几类纪律搬进了状态。战斗裁定会登记必须落账的义务，没结清的话 commit 整体打回；阵营时钟到期会在提交返回值里催办；悬疑钩子有数量上限（同时最多 2 个施压钩子），每次复现必须带新信息，禁止反复用同一扇门响吊人。

**压缩改成确定性截断。** 不再让 LLM 写摘要。事实都在引擎状态里，历史正文分层传给渲染端，丢上下文不再等于丢事实。

**`/fuck` 一键回滚。** 中断当前回合，回退到上一次玩家输入，丢弃废弃分支。结果不理想时可以直接回滚重试。

## 运行方式

主要测试组合：GPT-5.5 结算 + Fable 5 渲染。单模型跑过 Opus 4.5、DeepSeek V4 Pro、Gemini 3.1 Pro。子代理预设为 DeepSeek V4 Pro，可按自己的环境改。

系统挑的是工具调用纪律，不挑文笔。参数错可以打回重试；模型不愿用工具的话，系统就退回一张 prompt 卡。

推荐用函数调用比较稳定、愿意读工具报错的模型。那种一上来就写三千字正文、状态结算全靠猜的模型，跑这个会很痛苦。

Unix 系统本地运行：

```bash
pnpm install
./start.sh
```

第一次运行需要按自己的 pi 环境配置模型或登录。

项目会把运行配置隔离在 `.pi/agent/`，游玩记录在 `sessions/`。这些是本地东西，不要拿去分享。

## 适合谁

想要轻量开聊、角色陪伴、随手调 prompt 的话，这个太重了。它启动慢，规则多，还会因为状态错误打断模型继续写。

对下面这些事有感觉的话，可能会有兴趣：

- 酒馆卡能不能拆成更工程化的 agent；
- TYPE-MOON 这种设定密度很高的世界怎么做长线沙盒；
- 怎么减少“模型以为发生了但状态没有”的幽灵事实；
- 怎么让后台阵营在玩家休息时也合理行动；
- 怎么让叙事少靠气氛，多靠代价、状态和可交互后果推进。

这个实验品想试的方向：把角色卡做成一套本地运行的叙事系统，看长线游玩会不会更稳。
